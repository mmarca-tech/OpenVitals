# Feature Map

This is the canonical map from user-facing docs to navigation routes, dashboard tiles, and implementation directories. It should be updated when a feature adds a new route, tile, or owned directory.

This app is Flutter. Screens are widgets, view models are Riverpod `Notifier`s with `freezed` `*State` classes, dependency wiring is Riverpod providers, local persistence is drift, and navigation is go_router. The retired Kotlin implementation survives only in git history.

Source-of-truth anchors:

- Route paths and typed location builders: [`lib/navigation/app_routes.dart`](../../lib/navigation/app_routes.dart) (`AppRoutes`, `ActivityEntryMode`, `DashboardMetricId`)
- Route registration and metric fan-out: [`lib/navigation/app_router.dart`](../../lib/navigation/app_router.dart) (`goRouterProvider`, `metricScreenFor`)
- Dashboard read set: `DashboardMetric` in [`lib/domain/model/dashboard_query.dart`](../../lib/domain/model/dashboard_query.dart)
- Dashboard rings and stat tiles (title, accent, target route): [`lib/features/dashboard/application/dashboard_display.dart`](../../lib/features/dashboard/application/dashboard_display.dart)
- Manual entry widgets: `ManualEntryWidgetId` in [`lib/features/manualentry/presentation/manual_entry_screen.dart`](../../lib/features/manualentry/presentation/manual_entry_screen.dart)
- Settings sections: `SettingsSection` in [`lib/features/settings/presentation/settings_section.dart`](../../lib/features/settings/presentation/settings_section.dart)
- Object graph (the Hilt replacement): [`lib/di/providers.dart`](../../lib/di/providers.dart) and [`lib/state/app_providers.dart`](../../lib/state/app_providers.dart)

## Notes

- **One metric route, parametric screens.** `/metric/:metricId` is the only per-metric route. `metricScreenFor` ([`app_router.dart`](../../lib/navigation/app_router.dart)) dispatches the id to a feature screen, and the branch order is load-bearing: the calories aggregate claims `CALORIES_OUT` / `ACTIVE_CALORIES` / `BMR`, then nutrition, then the body aggregate, then the movement metrics, then heart/vitals, then cardio load, then an explicit tail (`WORKOUT`, `CAFFEINE`, `CYCLE`, `HYDRATION`, `MINDFULNESS`, `SLEEP`). There are **no** concrete per-metric screen classes such as `ProteinScreen` or `RestingHeartRateScreen`; the parametric screens are `ActivityMetricScreen`, `NutritionMetricScreen` and `HeartMetricScreen`. (`lib/features/vitals/presentation/vitals_screens.dart` does hold thin fixed-metric wrappers over `HeartMetricScreen`, but the router does not use them.)
- **`BODY_ENERGY` has no `/metric` screen.** It falls through `metricScreenFor` to `MetricScreen` → `PlaceholderScreen` ([`lib/features/dashboard/presentation/metric_screen.dart`](../../lib/features/dashboard/presentation/metric_screen.dart)). The dashboard's Body Energy tile does not go there: it links to `/daily_readiness/body_energy/:date`.
- **Dashboard tiles do not all route to `/metric`.** Body-composition tiles open `/body`, heart/vitals tiles open `/heart_vitals`, calorie tiles open `/calories`, nutrient tiles open `/nutrition`, and the cardio ring opens `/activity/cardio_load` — see `dashboard_summary_presentation.dart`.
- **`DashboardMetricId` vs `DashboardMetric`.** `DashboardMetricId` ([`app_routes.dart`](../../lib/navigation/app_routes.dart)) is the *route* vocabulary — the `SCREAMING_SNAKE_CASE` id carried by `/metric/:metricId`. `DashboardMetric` ([`dashboard_query.dart`](../../lib/domain/model/dashboard_query.dart)) is the *read* vocabulary — what `DashboardDataLoader` fetches, split into a fast `dashboardQuickMetrics` pass and a background pass.
- **There is no bottom navigation.** `/dashboard` is home; every other destination is pushed onto the root navigator with its own back-enabled app bar. `TopLevelDestination` in `app_routes.dart` is dead code left over from an abandoned bottom-nav design.
- **Five settings route constants are declared but never registered**: `settingsCalories`, `settingsCaffeine`, `settingsSleep`, `settingsBodyEnergy`, `settingsPermissions`. `_settingsSectionRoutes` registers one route per `SettingsSection` value only; those preferences are rendered as cards inside the eight section screens.
- Manual entry routes write explicit user-entered records to Health Connect. Dashboard and metric detail routes are read-oriented unless they open an explicit add/edit flow.
- There is **no metric summary cache**. drift ([`lib/data/local/open_vitals_database.dart`](../../lib/data/local/open_vitals_database.dart)) declares exactly one table, `beverages`; everything else that persists lives in `SharedPreferences`. The only derived-value cache is `BodyEnergyTimelineCacheStore`, and it is prefs-backed too. See [architecture.md](../engineering/architecture.md).
- Proposal pages live under [Proposals](../proposals/README.md) so current feature docs stay focused on implemented behavior.
